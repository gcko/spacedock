## Issues — Workflow

- **Duplicate-name Agent dispatch race.** Twice in this session I called `Agent()` with a name that was being held by an ensign that hadn't yet acknowledged its shutdown_request. The runtime allowed concurrent dispatches under the same name; the spawn-control rules don't refuse the second `Agent()` call. Result was duplicate spike runs (cycle-4 + cycle-5 of nwq, $10 of unnecessary spike budget; race-spawned ensign for 2y that did the AC-6 re-run after I'd waived it). Honest FO discipline lesson: shutdown_request + acknowledgment is two-step; new dispatches under the same name should wait for the ACK. The runtime's permissiveness here is a latent footgun.
- **`@`-include syntax does NOT resolve in `Agent()` prompt args.** Empirically verified (single throwaway probe, ~$0.50). Spawn-phase expansion is restricted to the agent definition's `skills:` frontmatter preload (broken for plugin agents per upstream `claude-code#25834`) and `@`-include resolution in skill bodies (working). Arbitrary `@`-paths in `Agent()` prompt args land as literal strings. Recorded in `docs/research/2026-05-21-agent-spawn-prompt-expansion.md`. This narrows the design space for rdt: file-pointer convention needs ensign-side parsing of `Skill(...); Read X; ...`, no platform shortcut available.

## Issues — Spacedock

- **`status --set` mod-block enforcement caught a legitimate rollback as a "terminal advancement."** When I tried to roll back a `status=implementation` → `status=ideation` transition (after a mid-stage captain redirect), the mechanism refused with "cannot advance to terminal" because clearing the worktree field looked like terminal-shape. The captain's intent was a rollback, not advancement. The guard was correct on the spirit ("don't advance to terminal without a PR") but wrong on this particular intent. Workaround: I used `git checkout --` to undo the failed `--set` call. Worth thinking about whether status transitions backwards (advance to earlier stage) should bypass the terminal-guard, or whether the FO should set fields in a different order.
- **`benchmark-token` rotates on OAuth schedule.** Working assumption "swap once and it persists" is wrong — OAuth access tokens have ~hourly refresh windows. Multi-hour spike or validation sessions hit auth-class failures (401 in our case). The harness's `apiKeyHelper` mechanism (mentioned in `claude --bare` help) might let the harness fetch fresh tokens on-demand from the keychain instead of reading a stale file. Worth a small follow-up entity; not blocking 2y/0x9.

## Observations

- **Cycle-1 ideation tends to over-specify the change surface.** Both 0x9 ("intake claimed YAML edit; actually Makefile") and 2y ("intake claimed YAML edit; actually Makefile" again — same pattern) had their territory corrected by ensign during ideation. The intake author (me, in both cases) hadn't walked the actual code path. Lesson: intake should be high-level intent + invariants; the ensign's territory walk should be expected to refine the change-surface, not just confirm it.
- **The spike-first discipline keeps paying off.** nwq's empirical spike falsified the prose-hardening hypothesis cheaply; 0x9's spike caught the unknown-unknown (`list-standing` returns paths, not the rendered block); 2y's spike confirmed sonnet works at 5/5. Every entity that ran a spike first got the design unstuck before the implementation surface had to absorb the question.
- **3 file-cycle dispatch ceremony.** The current FO dispatch pattern writes 3 tmp files per `Agent()` call: input JSON, helper stdout, prompt extract. The runtime adapter contract only prescribes `echo '<json>' | claude-team build | parse | Agent(prompt=output.prompt)` — 0 files. The 3-file pattern was derived as defensive engineering. The captain caught this mid-session; the rdt entity proposes a deeper compression (file-pointer-as-prompt) that's separate from this ceremony issue but composes with it. Worth noting in the FO runtime adapter that "0 tmp files" is the canonical form.
- **Pre-PR-#231 CI flakes were attributed to 0x9 on first inspection.** When PR #231's CI failed `test_fetch_on_demand_dispatch` on run 2, my first instinct was "real regression — the FO retry path bypasses claude-team build under the new contract." Captain pushed back ("is this flaky or genuine regression"). Walking the fo-log.jsonl carefully showed the first two retries WERE FETCH-shape; only the third was INLINED-MANUAL, and the cascade was an upstream 529 outage. Lesson: when a test fails on a PR that changed dispatch shape, walk the dispatch shape via fo-log.jsonl before attributing to the change.
